Architecture

operator-os is the product home for the Operator OS instantiation engine. It provisions new owner-specific instances of a multi-operator AI system through a three-slice pipeline:

Three-Layer Model

B — Template & Rewriterewrite-templateIdentity, operator role files, bridge files, path references — rendered with onboarding answers (25 sources + 5 bridges = 30 total)
C — User-Generatedgenerate-freshMemory, recent context, auth, projects — generated empty or from starter templates (13 surfaces)

Modules

Installer wired

manifest.js
Builds the installer execution manifest from user inputs + file matrix. Normalizes runtimes, resolves all target paths, classifies 305 surfaces into copyCore (31), templateSources (25), bridgeTemplates (5), safeScaffolds (6). Returns frozen manifest object.
Deps: runtime_selector, path_resolver, file_matrix
core_copier.js
Recursively copies Layer A (reusable-core) files from source vault to target. Pre-flight validation ensures all sources exist. Supports directory and file-level copy.
Deps: fs, path
template_placer.js
Places unrendered template source files and bridge templates to target locations with {{placeholders}} intact. Creates parent directories as needed.
Deps: fs, path
scaffold_generator.js
Generates safe starter config files from scaffold-safe templates. Verifies no forbidden source-machine paths leak into content.
Deps: fs, path
validator.js
5-checkpoint validation of Slice 1 output: structural presence, forbidden residue, forbidden artifacts, no owner generation, runtime gating.
Deps: runtime_selector

Onboarding wired

questionnaire.js
Validates raw onboarding input packet. 8 required fields, 16 optional fields with defaults. Normalizes runtime selection. Returns frozen packet.
Deps: runtime_selector
template_renderer.js
Renders runtime bridge templates (Codex, Gemini) into the vault root by resolving {{placeholders}} from onboarding answers. CLAUDE.md is handled by the source renderer. Verifies no unresolved markers remain.
Deps: template_utils
source_renderer.js
Rewrites vault source files (BOOT.md, ROUTING.md, etc.) replacing Viktor references with new owner identity. Ordered longest-first replacement to prevent partial matches.
Deps: fs, path
fresh_generator.js
Generates Layer C fresh surfaces: vault files (memory.md, identity.md, etc.) and runtime configs. Loads templates, resolves placeholders, validates cleanliness.
Deps: runtime_selector, template_utils, file_matrix
orchestrator.js
Coordinates the entire Slice 2 and Slice 3 pipelines. Slice 2: validate → render bridges → render sources → generate fresh → validate. Slice 3: connect accounts → pipeline config → bootstrap registry → first sync → voice profiler → validate.
Deps: all onboarding modules
validator.js
7-checkpoint validation of Slice 2: completeness, no Viktor residue, path integrity, boot chain, functional content, template cleanliness, runtime gating.
Deps: runtime_selector

Slice 3 — Account Setup & Sync wired

account_connector.js
Routes account auth through the connector registry and auth dispatcher instead of hardcoded branches. Handles now/later/mock/declared paths. Maps adapter results back to legacy return shapes for backward compatibility.
Deps: fs, path, registry, auth_dispatcher
pipeline_configurator.js
Generates pipeline_config.json from account connections and connector state. Populates owner, accounts, sync_engines, gmail_filters, workflow_tools, and connectors section (enabled, unsupported, deferred). Loads connector registry to resolve workflow tool selections. Verifies pipeline patches in 4 source files.
Deps: fs, path, registry
registry_bootstrapper.js
Creates fresh registry.json with project skeleton entries from user-supplied project categories. No Viktor project IDs.
Deps: fs, path
first_sync.js
Runs first pipeline sync after account connection. Planner-driven path uses connector sync adapters dispatched by the execution planner. Legacy fallback uses hardcoded CLI step list. Mock path for testing. Derivation (store_enrich, derive_all) runs after source sync succeeds.
Deps: fs, path, child_process, planner, registry, pipeline_configurator
voice_profiler.js
Analyzes captured sent emails and Basecamp comments to extract communication voice patterns. Outputs operator/voice.md with greeting/sign-off patterns, sentence structure, emoji frequency.
Deps: fs, path
slice3_validator.js
7-checkpoint validation of Slice 3: pipeline config, pipeline patches, no Viktor identity, fresh registry, first sync, today page, voice profile.
Deps: fs, path

Shared Utilities wired

runtime_selector.js
Normalizes runtime selection across 3 runtimes (Codex, Claude, Gemini). OpenClaw deferred. Pure utility — no state, no deps.
path_resolver.js
Resolves all target paths for vault (boot, routing, operators, etc.) and runtime (bridge, config) locations. Validates no forbidden roots leak.
Deps: runtime_selector
template_utils.js
Placeholder resolution ({{key}} → value) and unresolved marker detection. Pure utility.
file_matrix.js
Loads file-treatment-manifest.json and provides filtered views by treatment, owner phase, runtime, and scaffold safety. Central manifest accessor for all phases.

Pipeline Config wired

pipeline_config.js (project root)
Runtime accessor for pipeline_config.json. Provides getGmailAccounts(), getBasecampApiBase(), getCalendarConfig(), getDriveConfigDir(), getGmailFilters(). Returns null/empty when no config exists. No owner-specific fallback values.
Deps: fs, path

Connector Architecture wired

manifest_schema.js
Defines connector manifest schema and validates instances. Fields: id, display_name, category, capabilities, auth_adapter, sync_adapter, stage, priority, after, status. Exports VALID_STAGES, VALID_STATUSES, VALID_CATEGORIES.
registry.js
Discovers and loads connector manifests from adapters directory. Validates each manifest on load. Provides loadRegistry(), resolveAdapter(), getConnectorsByStage(), getConnectorsByCapability(). No connector is hardcoded into the registry.
auth_adapter.js
Defines the auth adapter contract: start_auth, finish_auth, refresh, revoke, status, required_secrets. Provides validateAuthAdapter() and createAuthResult() factory for standardized frozen result objects.
sync_adapter.js
Defines the sync adapter contract: validate_config, initial_sync, incremental_sync, healthcheck, normalize (required), discover (optional). Provides validateSyncAdapter() and createSyncResult() factory.
auth_dispatcher.js
Routes auth calls through the registry. dispatchAuth() resolves adapter and calls action. dispatchAllAuth() handles batch auth. buildGuidance() generates human-readable auth instructions. No hardcoded connector knowledge.
planner.js
Fixed four-stage execution planner: preflight, source_sync, derivation, post_sync. Deterministic ordering by priority, topological sort on after constraints, alphabetical tiebreak. Validates dependency graphs, rejects cycles and cross-stage deps. executeStages() drives live sync execution.
tool_resolver.js
Maps user workflow tool selections to connector state. resolveWorkflowTools() produces supported/unsupported/deferred status per tool. formatConnectorSummary() generates human-readable output lines. Pure logic, no stdin.
gmail adapter (auth.js + sync.js + manifest.json)
Production connector. Auth wraps gws CLI login and verification. Sync handles email, calendar, drive, sheets as internal sub-steps of a single connector. Category: email. Capabilities: email, calendar, drive, sheets. Stage: source_sync, priority 10.
basecamp adapter (auth.js + sync.js + manifest.json)
Production connector. Auth wraps Basecamp OAuth 2.0 flow with token refresh. Sync handles basecamp_sync. Category: project_management. Capabilities: tasks, messages, comments. Stage: source_sync, priority 20, after: gmail.

Design Rules

  • Installer places — onboarding renders. Non-overlap rule.
  • Manifest wins over other docs for file-level treatment decisions.
  • Zero Viktor residue in output. Validated by every slice.
  • Authority docs are build-time inputs, never shipped.
  • Pipeline config returns null/empty for unconfigured values. No owner-specific fallbacks.
  • Every credential is generated fresh — never copied from source.
  • Unsupported sync engines flagged as dev tasks, not install failures.
  • Adding a new connector requires only manifest + adapter files. No core edits.
