# Exhibition Lifecycle

If the question is about operator-wide identity or action boundaries, read `../../operator/CLAUDE.md`.
If the question is about recurring exhibition approval logic, read `approval-rules.md`.
If the issue is specific to one intake or active project, leave this file and go to the relevant project folder.
If the question is about the recurring shape of exhibition work, stay in this file.

This file defines the recurring lifecycle shape for exhibition projects.
It describes what tends to recur across exhibitions, not the facts of any one project.

## Core Rule

An exhibition project should be treated as a recurring operating pattern, not as a blank page each time.

Known exhibition structure creates a stronger prior, but it does not justify blind reuse.
Every exhibition still needs suitability judgment for its assets, timing, partners, and setup path.

## Canonical Exhibition Flow

The default recurring flow is:

`project detected -> intake reasoning -> asset strategy -> setup approval -> activation -> pre-production -> production execution -> install/opening -> live run -> closeout`

This means:
- exhibitions begin in `intake/`, not directly in `projects/`
- asset suitability should be judged before activation
- setup work is part of the project lifecycle, not a side detail
- activation should produce operational next actions, not just a status change

## Recurring Exhibition Stages

### 1. Intake And Classification

The exhibition is detected and identified as a likely exhibition project.

Typical questions:
- is this actually an exhibition project
- is it a new project, continuation, or merge case
- is the project identity accepted
- what evidence already exists

Typical outputs:
- project identity hypothesis
- type hypothesis
- missing information
- initial evidence set

### 2. Asset Strategy

The system evaluates the operating assets needed to run the exhibition.

Assets to reason about:
- milestone structure
- SOP
- production calendar
- Drive folder structure
- contract preparation path

For each asset, the system should recommend:
- `use_existing`
- `adapt_existing`
- `create_new`
- `defer`

### 3. Setup Preparation

Once the strategy is accepted, the system prepares the exhibition environment.

Typical setup outputs:
- Drive structure created or adapted
- production calendar created, copied, or adapted
- contract path identified or started
- references and links registered
- milestone structure linked or locked
- first setup actions created

Setup should be treated as real work with dependencies and blockers, not just metadata.

### 4. Activation

The exhibition becomes an active project when identity and setup direction are sufficiently resolved for execution to start.

Activation does not require every downstream detail to be complete.
It does require enough clarity that the project is now a real execution surface rather than an unresolved intake case.

### 5. Pre-Production

This is the preparation phase before the exhibition is physically or publicly live.

Typical recurring themes:
- scope alignment
- timeline structure
- production dependencies
- fabrication or logistics coordination
- approvals that shape downstream execution
- document and asset readiness

### 6. Production Execution

The exhibition moves through the operational work required to make it real.

Typical recurring themes:
- execution against the production calendar
- dependency management
- issue escalation
- coordination across people, vendors, and entities
- preserving continuity as changes happen

### 7. Install And Opening

The exhibition enters its highest-sensitivity execution window.

Typical recurring themes:
- final sequencing
- site or venue readiness
- opening-critical tasks
- time-sensitive communication
- fast correction when reality diverges from plan

### 8. Live Run

The exhibition is live, but the work is not over.

Typical recurring themes:
- active monitoring
- communications and relationship follow-up
- issue handling
- maintaining continuity across operational threads

### 9. Closeout

The exhibition exits the live phase and moves into completion and memory preservation.

Typical recurring themes:
- unresolved loose ends
- final documents and references
- durable decisions worth preserving
- context cleanup without losing useful history

## Exhibition-Specific Reality

Exhibition projects often combine:
- creative judgment
- commercial consequences
- relationship-sensitive communication
- production dependencies
- physical-world timing

That means exhibitions often look similar at a high level while differing materially in:
- timeline compression
- venue or geography
- partner model
- fabrication path
- contract shape
- stakeholder sensitivity

The recurring structure should guide the work, but never flatten these differences.

## What Belongs Here

This file should hold:
- recurring exhibition stages
- recurring operating expectations
- common structural logic
- the durable shape of exhibition work

This file should not hold:
- live task state
- approvals for a specific project
- one-off project exceptions that belong in a project folder
- runtime execution history
